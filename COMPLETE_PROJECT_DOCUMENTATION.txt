PROJECT OVERVIEW
================================================================================

Project Name: Court Judgment Predictor
Version: 1.0
Created: 2024-2025
Last Updated: December 04, 2025

DESCRIPTION:
The Court Judgment Predictor is an end-to-end machine learning application that
accepts free-text legal case facts and predicts multiple judicial outcomes: a
verdict label (e.g., Guilty / Not Guilty), primary IPC section, relevant law
labels (multi-label), and an estimated penalty in months. It also provides a
list of similar past cases via dense retrieval to support explainability.

PURPOSE:
- Provide prototype legal-assist tools for case triage and research
- Demonstrate multi-task transformer models in a legal domain
- Offer an auditable retrieval-backed explanation for each prediction

KEY FEATURES:
- Multi-task DistilBERT backbone with four prediction heads
- Multi-label prediction for relevant laws with class balancing
- Penalty estimation (regression) with scaling
- Dense retrieval using SBERT + nearest-neighbors (upgradeable to FAISS)
- FastAPI-based serving with a simple web UI and authentication routes
- Training utilities and synthetic dataset generator for reproducibility

SYSTEM COMPONENTS & INTERACTIONS:
- Ingest: `case_fact` text via API or UI
- Preprocess: cleaning and tokenization (AutoTokenizer)
- Model: DistilBERT backbone shared across heads
- Retrieval: SBERT encodes case text; nearest neighbor lookup returns similar
  case summaries
- Persistence: model artifacts in `model_artifacts/` and embeddings saved as
  `.npy` files


DATA
================================================================================

PRIMARY DATASET:
- `balanced_final_dataset_v6.csv` generated by `dataset_generator.py` —
  synthetic balanced dataset of Guilty/Not Guilty across 25+ IPCs.

EXPECTED COLUMNS:
- `case_id`, `case_fact`, `verdict`, `ipc_section`, `relevant_laws`,
  `penalty_months`, `similar_case_summaries`, `severity`

PREPROCESSING:
- Minimal cleaning currently: lowercase, remove newlines, normalize whitespace
- Tokenization: `AutoTokenizer` with `padding='max_length'`, `truncation=True`,
  `max_length=256`
- Recommendation: add Unicode normalization, NER-based PII masking, and
  canonicalization for law labels

LABEL ENCODING & SCALING:
- `LabelEncoder` for `verdict` and `ipc_section` (saved as joblib)
- `MultiLabelBinarizer` for `relevant_laws`
- `MinMaxScaler` for penalty scaled values


MODEL & TRAINING
================================================================================

MODEL ARCHITECTURE:
- Shared `DistilBertModel` backbone (local copy `distilbert-base-uncased`)
- Pool CLS token → Dropout(0.2) → Four linear heads:
  - Verdict: classification (CrossEntropy)
  - IPC: classification (CrossEntropy)
  - Laws: multi-label (BCEWithLogitsLoss with per-label pos_weight)
  - Penalty: regression (MSELoss)

LOSS COMPOSITION:
- total_loss = loss_verdict + loss_ipc + LAW_LOSS_WEIGHT * loss_laws + loss_penalty
- Default `LAW_LOSS_WEIGHT = 2.0` (tunable)

TRAINING CONFIGURATION (defaults in `train.py`):
- Batch size: 8
- Epochs: 4
- LR: 3e-5
- Warmup: 6% of total steps
- Max tokens: 256
- AMP: enabled on CUDA
- Seed: 42

DATA LOADER & WINDOWS SAFETY:
- `num_workers=0`, `pin_memory=True` for Windows compatibility

RETRIEVAL:
- SBERT: `all-MiniLM-L6-v2` used to compute `case_embeddings.npy`
- Index: scikit-learn `NearestNeighbors(metric='cosine')` saved to joblib
- Recommendation: migrate to FAISS/HNSW for scalability

MODEL CHECKPOINTS & ARTIFACTS:
- Saved to `model_artifacts/`: `model_state.pt`, encoders (`le_*.pkl`),
  `mlb_laws.pkl`, `scaler_penalty.pkl`, tokenizer files, `case_embeddings.npy`,
  `nearest_neighbors.joblib`


INFERENCE & API SPECIFICATIONS
================================================================================

GENERAL:
- `app.py` exposes `POST /predict` which accepts `{"case_text": "..."}` and
  returns JSON with predictions and similar cases.

ENDPOINT: POST /predict
- Input: `case_text` string
- Process: clean -> tokenize -> model forward -> decode labels -> retrieve
  similar cases -> build `case_summary`
- Behavior: short-circuit on `Not Guilty` (current `app.py` behavior)

STANDARD RESPONSE (successful):
```
{
  "verdict": "Guilty",
  "verdict_confidence": 0.93,
  "ipc_section": "302",
  "ipc_confidence": 0.89,
  "relevant_laws": ["302", "120B"],
  "penalty_months": 180.0,
  "similar_cases": [...],
  "case_summary": "..."
}
```

ERRORS & VALIDATION:
- Return consistent error responses with `success` / `error` fields for
  client consumption. Add `GET /health` to verify model+artifact readiness.

SECURITY & PRIVACY
================================================================================

THREAT MODEL:
- Adversaries include network eavesdroppers, malicious users, insiders with
  filesystem access, and future quantum adversaries if PQC not used.

CONTROLS & RECOMMENDATIONS:
- Move hardcoded secrets (JWT `SECRET_KEY`) into environment variables or a
  secret manager.
- Use HTTPS in production. Add authentication/authorization for API.
- Redact PII before logging or storing text inputs; consider NER-based
  redaction pipeline.
- Encrypt model artifact storage or use access-controlled storage.

AUDIT & EXPLAINABILITY:
- Store anonymized prediction records for auditing. Provide token-level
  insights via attribution methods (Integrated Gradients / LIME) for explain.

POST-QUANTUM CONSIDERATIONS:
- Not applicable to model predictions; if storing sensitive data, consider
  future PQC for key management where needed (e.g., signing artifacts).


PERFORMANCE & SCALING
================================================================================

PERFORMANCE GOALS:
- Cold-start load: < 15s on CPU; < 5s on GPU
- Single-request latency: CPU ~200–800ms, GPU ~10–100ms (model dependent)
- Throughput: scale via batching and multiple replicas

KNOWN BOTTLENECKS:
- Transformer inference cost, SBERT encoding, and sklearn nearest-neighbor
  lookups for large datasets

OPTIMIZATIONS:
- Export model to ONNX or quantize for CPU inference
- Use FAISS or vector DB for retrieval
- Implement request batching and async workers for throughput

TECHNICAL MAPPING — WHAT IS USED, WHERE, AND WHY
================================================================================
This section maps the principal technologies and components to the files/folders
in the repository and explains the rationale for choosing each.

- Python (3.10+): core language used across the repo (files: `train.py`,
  `inference.py`, `app.py`, `dataset_generator.py`, `preprocess.py`). Chosen for
  ecosystem maturity in ML (PyTorch/Transformers), rapid development, and
  portability.

- PyTorch (`torch`): used as the deep learning framework (files: `train.py`,
  `inference.py`, `app.py`). Rationale: flexible model definition, wide
  community support, efficient GPU acceleration, and ease of saving/loading
  model state dict (`model_state.pt`).

- Hugging Face Transformers (`transformers`): provides `AutoTokenizer` and
  `AutoModel` used by `train.py`, `inference.py`, and `app.py` to build and
  tokenize inputs for DistilBERT. Chosen for pre-trained backbones and
  interoperability.

- sentence-transformers (SBERT): used to compute dense case embeddings
  (`train.py` saves embeddings to `model_artifacts/case_embeddings.npy` and
  `local_sbert/`). Rationale: high-quality semantic embeddings for retrieval
  with compact models like `all-MiniLM-L6-v2`.

- scikit-learn: utilities used across training and retrieval (`train.py` and
  `app.py`): `LabelEncoder`, `MultiLabelBinarizer`, `MinMaxScaler`, and
  `NearestNeighbors`. Rationale: robust preprocessing utilities and simple k-NN
  for initial retrieval implementations.

- joblib: used to persist encoders, scalers and nearest-neighbors index
  (files: `model_artifacts/*.pkl`, `nearest_neighbors.joblib`). Rationale: fast
  serialization for numpy/scikit artifacts.

- numpy / pandas: data manipulation and storage (files: `train.py`,
  `dataset_generator.py`, arrays in `model_artifacts/`). Rationale: standard
  data science stack for tabular data and numerical arrays.

- FastAPI + Uvicorn (`app.py`, `auth.py`): lightweight REST API for serving
  predictions and frontend templates. Chosen for async-capable, production-ready
  serving with automatic OpenAPI docs.

- Jinja2 templates + `templates/` + `static/`: simple UI served by `app.py`.
  Rationale: minimal dependency for a working dashboard and static frontend.

- Tokenizer & local model directories (`distilbert-base-uncased/`,
  `local_sbert/`): local copies of models/tokenizers saved during training and
  retrieval setup (tokenizer saved to `model_artifacts/`). Rationale: avoid
  remote downloads at inference/serve time and ensure reproducibility.

- `dataset_generator.py`: programmatic synthetic data generation (output:
  `balanced_final_dataset_v6.csv`). Rationale: create balanced training data
  for prototyping and repeatable experiments.

- `preprocess.py`: centralized text cleaning helper invoked by training and
  inference code. Rationale: consistent minimal preprocessing across pipelines.

- `model_artifacts/`: artifact repo for model state (`model_state.pt`),
  encoders, scalers, embeddings, and index files. Rationale: single source for
  loading production artifacts in `app.py` and `inference.py`.

- NearestNeighbors (scikit-learn) vs FAISS (future): current implementation
  uses scikit-learn's `NearestNeighbors` for simplicity and local small-scale
  retrieval (`train.py` + `app.py`). FAISS or HNSW is recommended for
  production-scale retrieval due to speed and memory efficiency.

- Authentication (`auth.py`) & security libs: `python-jose`, `passlib` used in
  the auth module (folder/file: `auth.py`, `users.json`). Rationale: provide a
  simple password-hash and JWT flow for demo authentication; must move
  `SECRET_KEY` to environment in production.

- Tests & CI (recommended additions): place unit tests under `tests/` (e.g.,
  `tests/test_preprocess.py`, `tests/test_api_smoke.py`) and add a GitHub
  Actions workflow to run linting and tests. Rationale: ensure basic
  correctness on PRs and maintain quality.

This mapping should help developers quickly find where a capability is
implemented in the codebase and understand the design rationale for each
technology choice.
